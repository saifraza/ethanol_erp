# Ethanol Supply — Job Work Invoice Linking Postmortem (2026-04-11)

## What Happened

19 MASH BIO-FUELS liftings (April 6-11) showed "GEN INVOICE" despite invoices already existing with IRN + EWB generated. The ethanol supply page appeared broken:
- No invoice numbers on liftings
- No DCH/GP document buttons
- No EWB PDF access
- 50+ ghost trucks showing as "Loading at Site"
- Active trucks from gate entry not appearing under the contract

## Root Cause Chain

### 1. Invoice Created But Never Linked (PRIMARY BUG)

The `POST /:id/liftings` endpoint (ethanolContracts.ts:254) auto-creates invoices inside a `setImmediate` fire-and-forget callback (line 316):

```typescript
if (contract.autoGenerateEInvoice && rate && amount && contract.buyerGst) {
  setImmediate(async () => {
    // creates invoice, then links:
    await prisma.ethanolLifting.update({ data: { invoiceId: inv.id } }); // line 347
    // if ANYTHING before this line throws, invoiceId stays null
    // error is caught and logged but never surfaced to user
  });
}
```

The invoice creation succeeded (timestamps match within 6ms), but the `ethanolLifting.update` to set `invoiceId` silently failed. The `catch` block only logs to console — no retry, no alert, no PlantIssue.

**Why it failed silently**: The `setImmediate` pattern means the HTTP response is already sent. Any error in the callback is invisible to the user. The invoice exists but is orphaned.

### 2. DispatchTrucks Not Linked to Contract

The `ethanolOutbound` weighbridge handler (handlers/ethanolOutbound.ts:119) auto-creates DispatchTruck records when factory syncs weighments, but it never set `contractId` from the `cloud_contract_id` field — even though the factory gate entry sends it and the Zod schema accepts it.

Result: All DispatchTrucks had `contractId: null`, so the supply-summary endpoint's `activeTrucks` query (`where: { contractId: contract.id }`) returned zero.

### 3. Ghost Trucks from Pre-Weighbridge Era

~115 DispatchTruck records from March 10 - April 5 were stuck at `GATE_IN` status. These were manual entries from before the weighbridge integration — trucks were dispatched but status was never updated. When we linked them to the MASH contract by party name, they all showed up as "Loading at Site".

### 4. Duplicate EWB PDF Route

Two `GET /:id/liftings/:liftingId/ewb-pdf` routes existed:
- Line 424: Serves uploaded PDF binary (`ewbPdfData`)
- Line 1126: Generates PDF from EWB data fields

Express uses the last registered route, so uploaded PDFs (from manual job work EWB uploads) were never served — the second route tried to generate from data and failed with 404.

### 5. Missing Challan Numbers

Liftings created manually (not through weighbridge release) never got `challanNo` because it's only generated by `nextCounter(tx, 'DCH/ETH')` in the release flow.

## Permanent Rules

### RULE 1: Never Use setImmediate for Critical Writes
If an operation MUST succeed (like linking invoiceId), do it **in the same request** or **in a transaction** — not in fire-and-forget. `setImmediate` is only for truly optional side-effects (notifications, logging).

**FIXED (2026-04-11)**: Invoice creation + linking now runs in a `$transaction` in the main request flow. Only IRN/EWB generation (external API calls) stays in `setImmediate`. If the invoice+link fails, the transaction rolls back — no orphaned invoices.

### RULE 2: Every DispatchTruck Must Have contractId
When creating a DispatchTruck from any source (gate entry, weighbridge sync, manual), always set `contractId`. The contract is the organizing principle — without it the truck is invisible in the supply view.

### RULE 3: DispatchTruck Status Must Be Updated
When a lifting is created from a truck (whether via release or manual), the truck status MUST move from GATE_IN → DISPATCHED/RELEASED. Never leave completed trucks at GATE_IN.

### RULE 4: Don't Bulk-Link Old Data Without Date Filtering
When backfilling `contractId` on DispatchTrucks, ALWAYS filter by date. Old manual entries from before the system was live are ghost records — linking them pollutes the active view.

### RULE 5: EWB PDF — Uploaded Binary Takes Priority
For job work contracts, EWB is generated manually on ewaybillgst.gov.in and uploaded as PDF. The ewb-pdf endpoint must check `ewbPdfData` (uploaded binary) first, then fall back to generated PDF.

### RULE 6: Challan Numbers Must Be Generated at Lifting Creation
If a lifting is created outside the weighbridge release flow (manual, import, API), it still needs a challan number. Generate via `nextCounter(tx, 'DCH/ETH')`.

## Architecture Note: Two Paths to Liftings

There are two ways liftings get created — both must produce identical data:

**Path A: Weighbridge Release** (ethanolGatePass.ts POST /:id/release)
- Creates DispatchTruck → weighment → release → creates Lifting + Invoice + Challan + GatePass
- Atomic transaction, all fields populated
- This is the "correct" path

**Path B: Manual Lifting** (ethanolContracts.ts POST /:id/liftings)
- Creates Lifting directly, then optionally creates invoice in setImmediate
- No DispatchTruck, no challan/gatePass numbers
- Missing: contractId on truck, challan generation, invoice linking reliability

**TODO**: Path B should be brought to parity with Path A — create DispatchTruck + generate challan/gatePass numbers inline. Or better: eliminate Path B entirely and route all liftings through the weighbridge flow.
